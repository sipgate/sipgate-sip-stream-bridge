--- src/logger/index.ts ---

Create pino root logger and child logger helper. Import: import pino, { type Logger } from 'pino'

Root logger:
```typescript
export const rootLogger: Logger = pino({
  level: process.env.LOG_LEVEL ?? 'info',
  base: { service: 'audio-dock' },
});
```

Child logger helper (establishes OBS-01 per-call context pattern):
```typescript
export function createChildLogger(bindings: {
  component: string;
  callId?: string;
  streamSid?: string;
}): Logger {
  return rootLogger.child(bindings);
}
```

Note: LOG_LEVEL is read directly from process.env here because pino initialises before config module imports it through the module graph. This is intentional — if config load fails, the logger already running level is still correct.

--- src/index.ts ---

Entry point. Import order matters: config must be first (it calls process.exit on failure before logger is used), then logger, then start the application.

```typescript
import { config } from './config/index.js';
import { createChildLogger } from './logger/index.js';

const log = createChildLogger({ component: 'main' });

log.info(
  { event: 'startup', sipUser: config.SIP_USER, sipDomain: config.SIP_DOMAIN },

// Phase 2 will call createSipUserAgent here.
// For Phase 1, keep the process alive so Docker can observe it running.

// process stays alive via event loop; no explicit blocking needed.
```

Do NOT use setInterval or any dummy timer just to keep the process alive — Phase 2 will have real async activity. For Phase 1 the process will exit immediately after logging. That is acceptable; the goal is to verify config validation works, not to hold a long-running process.
