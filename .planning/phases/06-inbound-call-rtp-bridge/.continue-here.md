**Bug 4 — Jitter** (internal/bridge/session.go):
Root cause: `rtpToWS` uses a 100ms `SetReadDeadline` loop. When the goroutine is briefly
de-scheduled (OS/GC), the deadline fires, goroutine does a `continue`, resets deadline, and
reads ALL buffered UDP packets in rapid succession → burst of 0ms iat packets at the listener.

Fix:
- Remove the `SetReadDeadline` / timeout loop from `rtpToWS` entirely
- Use a plain blocking `rtpConn.ReadFromUDP(buf)` (no deadline)
- For clean shutdown: in `run()`, after `<-sessionCtx.Done()`, call `rtpConn.Close()`
  BEFORE `s.wg.Wait()`. This unblocks rtpToWS (gets error → ctx check → return silently).
  It also causes rtpPacer WriteTo to fail → s.cancel() (harmless, already cancelled) → return.
- `defer rtpConn.Close()` already registered — double-close is harmless.
- rtpToWS error handler: add `if ctx.Err() != nil { return }` check (same pattern as wsToRTP).
- Also add `strings` import to manager.go for `strings.ReplaceAll`.

<decisions_made>

- packetQueue/rtpPacer architecture: wsToRTP queues 160-byte PCMU frames; rtpPacer sends at 20ms intervals via ticker — handles blobs up to ~1MB per Twilio spec
- rtpPacer sends PCMU silence (0xFF × 160) when queue empty — required for NAT traversal (punches NAT hole immediately at call start); keeps NAT mapping alive
- wsToRTP shutdown: `wsConn.SetReadDeadline(time.Now())` in run() after ctx done — unblocks blocking WS read without corrupting frame state
- rtpToWS shutdown (planned): `rtpConn.Close()` instead of 100ms deadline loop — eliminates burst-jitter from deadline timeouts
- callSid in Twilio protocol = CA-prefixed token (NOT the SIP Call-ID); SIP Call-ID goes in customParameters.sipCallId
- 180 Ringing must precede WS dial; WS dial must precede 200 OK (ring → connect → answer)

</decisions_made>
