## Bug 3 (CANCEL) — FIXED
- Root cause: sipgo calls `tx.TerminateGracefully()` immediately after the INVITE handler
  returns. With only provisional responses (100+180), it calls `tx.Terminate()` → closes
  `tx.Done()` → goroutine's `RespondSDP` hits `WriteResponse`'s `<-tx.Done()` check →
  returns `ErrTransactionTerminated` ("transaction terminated"). No CANCEL involved.
- Fix (commit `5553b01`): Call `dlg.RespondSDP(sdp)` SYNCHRONOUSLY in `onInvite` BEFORE
  the handler returns. Each INVITE runs in its own goroutine (`go txl.handleRequestBackground`),
  so blocking until ACK is safe. After ACK, launch goroutine for WS+bridge only.
- `StartSession` no longer calls RespondSDP — call is already answered when it runs.
- Port released via `ReleasePort()` on RespondSDP failure in onInvite, or via defer in
  StartSession on success.
- **Live test passed**: Call is now answered, listener sees connected+start events.

## Bug 4 (Jitter) — TCP_NODELAY attempted, NOT fixed
- Identified `ws.WriteFrame` makes 2 separate `write()` syscalls: `WriteHeader` (8B) then
  payload (350B). Hypothesis: Nagle algorithm holds payload write waiting for header ACK.
- Fix applied (commit `c30e804`): `TCP_NODELAY` set on WS TCPConn after `ws.Dial`.
- **Result**: Stall pattern unchanged — still ~100ms stall every ~26 packets (520ms period).
- **Conclusion**: Stall is in `ReadFromUDP`, not in `writeJSON`. TCP_NODELAY confirmed that
  the write path is not the bottleneck. The RTP packets themselves are not arriving for ~100ms.

## Bug 4 (Jitter) — root cause still unknown, fix needed

**Current hypothesis**: The ~100ms stall every ~26 packets (~520ms period) is happening in
`ReadFromUDP`. Possible causes:

1. **Sipgate/phone RTP sender**: The phone or sipgate's RTP proxy has a periodic batching
   behavior — holds packets for ~100ms every ~500ms then sends them in a burst. This would
   be upstream of our code and outside our control.

2. **Go GC or scheduler**: Some periodic Go runtime event (GC, goroutine preemption) blocks
   `rtpToWS` for 100ms — though Go GC pauses are typically <1ms with concurrent GC.

3. **OS/network interface**: macOS kernel UDP receive path has some periodic stall.

**Next diagnostic steps**:
- Add RTP timestamp logging inside `rtpToWS` to log the RTP header timestamp alongside the
  wall-clock arrival time. If RTP timestamps show consistent 160-sample increments but wall
  time jumps 100ms, packets were delayed in transit (network/sender issue). If RTP timestamps
  also show gaps, packets were lost/reordered.
- Alternatively: look at whether the phone's codec or sipgate's gateway does DTX/VAD.
- Check if the pattern exists with a different SIP client (e.g., softphone).

**Potential fixes if network-side**:
- Implement a simple RTP playout buffer: buffer packets by RTP timestamp, forward at constant
  rate (20ms/packet). Adds latency (~40ms) but smooths jitter for the WS consumer.
- Use `SetReadDeadline` on the UDP socket with a short timeout (but this caused burst-jitter
  before — see comment in rtpToWS).

**Other open items from BUGS.md**:
- `User-Agent` header not set in SIP responses (old Node.js sent `audio-dock/0.1.0`)
- `To` field format in start event differs from old impl (old: `sip:4921187979819@192.168.6.74:5060`)

</remaining_work>

<context>
session.go:rtpToWS is a tight loop: ReadFromUDP → Unmarshal → base64 → JSON → writeJSON.
With TCP_NODELAY, writeJSON completes in <1ms. Therefore any >20ms delay in the loop is
from ReadFromUDP blocking waiting for the next RTP packet.

The stall is exactly every 26 packets (26 × 20ms = 520ms). This regularity strongly
suggests a periodic cause, not random network jitter. Classic random network jitter would
produce a statistical distribution, not a clock-like pattern.

Key files:
- internal/bridge/session.go: rtpToWS goroutine (ReadFromUDP loop)
- internal/bridge/ws.go: dialWS (TCP_NODELAY set here), writeJSON
- internal/sip/handler.go: onInvite (RespondSDP now synchronous)
- internal/bridge/manager.go: StartSession (no more RespondSDP)
- jitter.log: latest timing log showing the stall pattern
</context>
