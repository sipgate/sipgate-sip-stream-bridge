Root cause of Bug 3 (CANCEL immediately after 180) identified and fixed.

<completed_work>

- Root cause found: sipgo's `NewResponseFromRequest` copies `Record-Route` headers
  from the INVITE into ALL responses including provisional 180/100. SIP clients
  (softphone + mobile) see `Record-Route` in 180 Ringing and immediately send CANCEL.
  The old Node.js / SIP.js implementation did NOT include Record-Route in 180.
- Also: sipgo's `dlg.Respond(180)` adds a `Contact` header automatically.
  Old implementation had no Contact in 180 either.
- Fix 1 (handler.go): Build 100 Trying and 180 Ringing manually via
  `siplib.NewResponseFromRequest`, call `ringing.RemoveHeader("Record-Route")`,
  send via `tx.Respond` directly (bypasses dlg.Respond which adds Contact).
- Fix 2 (manager.go): Moved `RespondSDP(200 OK)` BEFORE the WS dial to eliminate
  the CANCEL race window. WS dial now happens after the call is answered.
  If WS fails after answering: `dlg.Bye()` instead of 503.
- Fix 3 (handler.go): `onCancel` now responds 200 OK to retransmissions to stop
  sipgate from retransmitting CANCEL indefinitely.
- All tests pass with -race.

</completed_work>

<decisions_made>

- Record-Route in provisional responses (100/180) causes SIP clients to send CANCEL
  immediately — stripped via RemoveHeader("Record-Route") before tx.Respond
- Contact header in 180 omitted (use tx.Respond directly, not dlg.Respond which
  adds Contact automatically) to match old Node.js behavior
- RespondSDP (200 OK) moved before WS dial: eliminates race window where sipgo's
  CANCEL goroutine could win fsmMu before our goroutine completed the WS handshake
- WS failure after answering → BYE (not 503, since 503 impossible after 200 OK)
- onCancel responds 200 OK to stop retransmission loops

</decisions_made>

<context>
The investigation traced through sipgo v1.2.0 source (transaction_layer.go,
transaction_server_tx_fsm.go, dialog_ua.go, dialog_server.go, dialog.go) to
understand the CANCEL handling FSM, fsmMu races, and provisional response building.

Key sipgo internals:
- NewResponseFromRequest copies Record-Route (line 207 in sip/response.go)
- dlg.Respond(180) → WriteResponse → adds Contact if missing
- CANCEL handled atomically under fsmMu: tx.Receive → actCancel → endWithCause →
  dialog context cancelled → FSM → inviteStateCompleted (487 sent)
- Once FSM in Completed, server_input_user_2xx is ignored → can't answer
- tx.Done() closes only after Timer_H (32s), not immediately on CANCEL

The real fix was the Record-Route discovery from comparing old-good.log vs new-bad.log
SIP captures the user provided.
</context>
