What is it?

Twilio out, sipgate in. Your code stays.

Using Twilio Media Streams for voice bots or real-time audio? The SIP Stream Bridge replaces Twilio - not your code. It emulates the Twilio WebSocket interface and routes calls over sipgate SIP trunking.

Same protocol, same events, same payloads. Your code keeps running - only Twilio is gone.

Your backend
speaks Twilio Media Streams (WebSocket)
↓
SIP Stream Bridge
Protocol translation
↓
sipgate
SIP trunking (INVITE, RTP/SRTP)

Your backend sends WebSocket frames to the bridge.
The bridge maps them to standard SIP and connects to sipgate.

Why switch?

What sipgate does better.

🏢
Audio never leaves the EU
As a US company, Twilio is subject to the CLOUD Act - regardless of server location. The bridge runs self-hosted on your own machine, telephony over a German carrier: your audio touches neither Twilio nor a US cloud. No transatlantic detour, no CLOUD Act.
Self-hosted · CLOUD Act-free
🔒
GDPR - without the paperwork
No data-processing agreement with a US company. No standard contractual clauses. sipgate is based in Germany - so GDPR isn't a legal project, it's the default.
💶
Fair pricing, no middleman
Inbound calls are free with sipgate - no per-minute inbound charge and no Media Streams surcharge like Twilio's. Local landline and mobile numbers directly from sipgate, with no markup from a US middleman. The bridge itself is free and open source.
🤝
Support that speaks your language
sipgate has been around for over 20 years. Technically skilled support comes first. From techies, for techies.
🔓
100 % Open Source
Source code on GitHub. No vendor lock-in, no black box. Fork it, adapt it, send PRs - contributions welcome.

The switch

Switching is this easy.

1
Create .env
SIP_USER=1234567t0 SIP_PASSWORD=meinPasswort SIP_DOMAIN=sipconnect.sipgate.de SIP_REGISTRAR=sipconnect.sipgate.de WS_TARGET_URL=ws://mein-backend.example.com SDP_CONTACT_IP=1.2.3.4

SIP credentials from your sipgate trunking account. Don't have one yet? → sipgatetrunking.de

2
Start the container
# Go version (recommended) docker run \ --env-file .env \ --network host \ ghcr.io/sipgate/sipgate-sip-stream-bridge-go:latest

One Docker command. ~10 MB image, cold start in milliseconds.

3
Done ✓

FAQ

Frequently asked questions.

Do I have to rewrite my code?
+
No. The bridge implements the Twilio Media Streams protocol. Your backend won't notice a difference.
Do I need a sipgate account?
+
Yes - you need a sipgate trunking connection. Available at sipgatetrunking.de. The bridge itself is free.
Is this really GDPR-compliant - and what about the CLOUD Act?
+
Yes. The bridge runs self-hosted on your own infrastructure - the audio data never leaves your machine and runs over a German carrier. sipgate is based in Düsseldorf. Unlike US providers, there's no CLOUD Act risk (which binds US companies regardless of server location); SCCs and a separate DPA aren't needed.
What does the bridge cost?
+
The bridge is free and open source. Self-hosted, it runs on the smallest cloud server (1 vCPU, 512 MB RAM). You only pay for your sipgate trunking connection.
How many calls can the bridge handle simultaneously?
+
Default: up to 100 concurrent calls, configurable. Runs stably for weeks without a restart.
Does it work without Linux?
+
For production we recommend Linux with --network host. Locally on macOS or Windows, Docker Desktop works too.
